COUNT(), COUNT(*) and COUNT(1)
Use either COUNT(field) or COUNT(*), and stick with it consistently



Use count(*) for counting

=====================

For counting rows we could have this:



SELECT COUNT(emp) FROM emp



Counting with null

================

It is not a good practice to make a field nullable if its name matches the table name. Say you have values 'Banana', 'Apple', NULL, 'Pears' on fruit field. This will not count all rows, it will only yield 3, not 4



SELECT count(fruit) FROM fruit



count(1): The one-trick pony

=======================

In particular to COUNT(1), it is a one-trick pony, it works well only on one table query:



SELECT COUNT(1) FROM tbl

But when you use joins, that trick won't work on multi-table queries without its semantics being confused, and in particular you cannot write:



-- count the subordinates that belongs to boss

SELECT boss.boss_id, COUNT(subordinate.1)

FROM boss

LEFT JOIN subordinate on subordinate.boss_id = boss.boss_id

GROUP BY boss.id



Summary

========

COUNT(*) counts the number of rows

COUNT(1) also counts the number of rows

GREATEST vs. MAX()
At first glance, the MySQL MAX() and GREATEST() functions perform a similar operation. They both return the maximum value from a range of values. However, there’s a difference between these two functions.


The best way to see the difference between these two functions is to compare their syntax.



Syntax

The syntax for each function goes like this:



MAX([DISTINCT] expr) [over_clause]

GREATEST(value1,value2,...)


So already, they look completely different. The MAX() function accepts the DISTINCT keyword as well as an OVER clause (and the GREATEST() function doesn’t).



However, the key difference between these two functions is in the accepted argument/s. Specifically:



- MAX() accepts one argument

- GREATEST() accepts multiple arguments


So MAX() is typically used to return the maximum value in a column in a database. The table could contain many rows, but this function returns the one with the maximum value.



GREATEST() on the other hand, returns the maximum-valued argument from the list of arguments passed to it. So you could pass say, 3 arguments to this function and it will return the one with the largest value.

=============

SELECT MAX(Population) AS 'Result'

FROM City;

GREATEST() function

=====================

The GREATEST  function returns the “greatest” or “largest” value from the list of expressions.



GREATEST ( value_1, [value_n] )



SELECT GREATEST(1, 5, 9) AS 'Result';



+--------+

| Result |

+--------+

|      9 |

+--------+

---Json / JsonB Data type-----------

--Create table With JsonB

create table books(

book_id serial primary key,
book_info jsonb
)

select * from books

--Insert Json Data

insert into books(book_info) values
('{"title":"Book Title2","author":"Author2"}'),
('{"title":"Book Title3","author":"Author3"}')

--select all json record
select * from books

--Using Selectors (-> and ->>)
--1.. ->  this operator will return the JSON Objects field as a field in quotes

--Get all json column data...

select book_info from  books

--Get a particular field of json data

select book_info -> 'title'  from books  -- This value will result in double quotes by using ->

--2.. ->> this will return  the field value as in text not in double quotes

select book_info ->> 'author' from books

--we can get multiple field at time from json object

select book_info ->> 'author',book_info ->>'title' from books


--Filtering json data by Where clause

select 
 book_info ->>'title' as "title",
 book_info ->>'author' as "author"

from books where book_info ->> 'author' = 'Author2'


--Update Json Record into table...........
##############################################

--Lets insert some data

insert into  books(book_info) values
('{"title":"Title10","author":"author10"}')

 -- || it will add new field as well as it will update

GI Index :-
 Gi means generalized inverted Index.

--3664 kB for 20k record

/*
--Challanges With GIN Index :-
 Depending on the complexity of data maintaining GIN Indexes can be expensive....
 
 Creation of GIN Index consumes time and resources as the index has to search through the whole documents
  to find the key and theri row id..
 
 It Can be even more challanging if the GIN indexs is bolated..
 Also, the size of GIN Index can be vary big based on 	the data size and its complexity
*/


--There is better Index 
create index idx_gin_contacts_docs_body_cool on contacts_docs Using GIN(body jsonb_path_ops)


--Check Size of index............result in KB.......

select pg_size_pretty(pg_relation_size('idx_gin_contacts_docs_body_cool'::regclass)) as index_name

--Now size is 2512 kB for 20k record

--Now create the index on the specific json key
create index idx_gin_contacts_docs_body_fname on contacts_docs Using GIN((body->'first_name') jsonb_path_ops)

----CREATE INDEX-----------------
##########################################

/*

We Can Put the index on table column and columns

Why Can't be put the index on all columns ?
 Too Many index will result in slow insert/delete/update Operations (High Costs)

PostgreSQL supports indexes with up to 32 indexes.....

TWO Main Basic index Type :- 

INDEX  :- Create a index on only values of a column and columns 

UNIQUE INDEX :- create and index on only UNIQUE Value on column or columns


SYNTAX :-

CREATE INDEX index_name ON table_name (col1,col2.....)

CREATE UNIQUE INDEX index_name ON table_name(col1,col2...)

--Naming Convention of Index :---

CREATE INDEX idx_table_name_column_name

CREATE UNIQUE INDEX  idx_u_table_name_column_name


*/


-- Create Index On single column on order table

It is important to note that when creating multi column indexes we should always place the most slective

--INDEX TYPES :-
#######################

/*

B-TREE :-
##################

Default Index.. 

Self Balancing tree
 -Select,Insert , Delete, and sequential access in logarithmic time..

Can be used for most operators and column types.

supports the UNIQUE condition and

Normally used to build the primary key indexes

Uses when columns involves following operators

 < , > , <= , >= , BETWEEN , IN , IS NULL , IS NOT NULL


Use When Pattern Matching for LIKE base queries....

 customer_id like'12%'

--One Drawback Of B-Tree Copy the whole columns values into the tree structure..

*/


------Hash Index-------------
#######################################

/*

For Equality Operators (Only Simple equality comparision =)

Not for range or disequality operators.

Larger than b-tree index..


SYNTAX :-

CREATE INDEX index_name ON table_name USING Hash(column_name);

*/

--Create hash index......

CREATE INDEX idx_order_order_date ON orders USING hash(order_date)

--Use 
explain Select * from orders where order_date = '2020-02-02'

-----Expression Index------------
--#########################################

/*

1. An Index will get Created on expression  ex :-
 UPPER(first_name),
 LOWER(last_name)
 MIN(column)
 MAX(Column)
 EXTRACT(day from Date Column)

2. PostgreSQL Will consider ing that index when the expression that defines the index appears in the 
  --Where Clause or
  in the order by clause of the query statement

3. Very expensive Indexes
   PostgreSQL has to evaluate the expression for Each Rows when it is INSERTED or UPDATED and use the result for indexing.

SYANTAX :-

CREATE INDEX index_name ON table_name(expression)

*/

CREATE INDEX idx_express_t_date_created_date ON orders(EXTRACT(day from created_date));

select * from orders where EXTRACT(day from created_date) = 2
